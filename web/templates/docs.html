1. Overview

Duckway is an API proxy that manages real API keys centrally. AI agents and other clients never see real keys — they use placeholder keys that look identical in format but are swapped for real keys by the proxy.

Key benefits:

  • No key leakage — agents only have fake keys; real keys never leave the server
  • Fine-grained permissions — restrict which endpoints, models, and rate limits each client can use
  • Approval workflow — require admin consent before a key is first used
  • Load balancing — rotate across multiple API keys to avoid rate limits
  • Canary tokens — detect if a client machine is compromised
  • Audit trail — log every proxied request

The entire server is a single Go binary (~19MB) with SQLite embedded. No external database, no npm, no runtime dependencies. Admin panel uses Go templates + HTMX.

3. Quick Start

Server

# Start server (first run generates admin password)
./scripts/dev.sh --port 8080

# Or directly:
go run ./cmd/server/ --port 8080

Initial Setup (this panel)

  1. Login with the generated credentials shown in the terminal
  2. Default services (OpenAI, Anthropic, GitHub, Discord, Telegram) are pre-configured
  3. Go to API Keys → add your real API keys for each service
  4. Go to Clients → register a client machine → save the token
  5. On the client card, click Assign Key to generate placeholder keys

Client Machine

# Register with server (interactive)
duckway init

# Or non-interactive with token from admin panel:
printf "http://server:8080\nmy-laptop\n1\n<token>\n" | duckway init

# Sync placeholder keys
duckway sync

# Show keys as env vars
duckway env
eval $(duckway env)

# Check status
duckway status

4. Services & API Keys

A Service represents an upstream API (e.g., OpenAI, GitHub). Each service has:

NameSlug used in proxy URL: /proxy/{name}/...
Upstream URLWhere requests are forwarded (e.g., https://api.openai.com)
Auth TypeHow to inject the key: bearer, header, or query
Auth HeaderHeader name (e.g., Authorization, x-api-key)
Auth PrefixValue prefix (e.g., Bearer , Bot )
Key PrefixReal key prefix for format matching (e.g., sk-, ghp_)
Key LengthExpected key length for validation

API Keys are your real credentials, stored encrypted (AES-256-GCM) in the database. They are never exposed to clients — only the server uses them when proxying requests.

When adding a key, the input field shows the expected format and validates prefix + length before submission.

5. Client Setup

Each AI agent machine registers as a Client. Registration generates a pre-shared token (shown once — save it).

Registration Flow

  1. Admin creates client in Clients page → gets a token
  2. On the agent machine: duckway init with the token
  3. Client config saved to ~/.duckway/config.yaml
  4. Admin assigns API keys to the client → placeholder keys auto-generated
  5. Client runs duckway sync → keys written to ~/.duckway/keys.env

Docker Client

# Build and run
docker build -f Dockerfile.client -t duckway-client .
docker run -d --name duckway-client --network host duckway-client
docker exec -it duckway-client /bin/bash
# Then: duckway init

6. Placeholder Keys

Placeholder keys are fake credentials that look real. They match the format of the actual API key:

ProviderReal FormatPlaceholder Format
OpenAIsk-proj-ABC...sk-proj-dw_a8f3b1...
Anthropicsk-ant-ABC...sk-ant-dw_7e2f9a...
GitHubghp_ABC...ghp_dw_9c3e5f...

The dw_ marker identifies them as Duckway placeholders. Same prefix, same length — API client libraries accept them without validation errors.

Each placeholder is bound to one client + one API key (or key group). Assign them from the Clients page.

Clients sync these via duckway sync → stored in ~/.duckway/keys.env:

OPENAI_API_KEY=sk-proj-dw_a8f3b1c2d4e5...
ANTHROPIC_API_KEY=sk-ant-dw_7e2f9a1b3c...
GITHUB_TOKEN=ghp_dw_9c3e5f7a1b2d4e...

7. Proxy — How It Works

The proxy is a reverse proxy at /proxy/{serviceName}/. When an agent makes a request:

# Agent sends:
POST http://duckway-server/proxy/openai/v1/chat/completions
X-Duckway-Token: <client-token>
Content-Type: application/json
{"model": "gpt-4o", "messages": [...]}

# Server does:
1. Authenticate client via X-Duckway-Token
2. Look up service "openai" → upstream https://api.openai.com
3. Find placeholder key for this client + service
4. Resolve placeholder → real API key (decrypt from DB)
5. Check approval status
6. Check permissions (endpoint, method, body constraints)
7. Forward to: POST https://api.openai.com/v1/chat/completions
   with: Authorization: Bearer sk-real-key-here
8. Stream response back to agent
9. Log request

The agent never sees the real key. The proxy handles auth injection based on the service config (Bearer, header, query param).

8. Permission System

Each placeholder key can have a permission config (JSON) that restricts what the client can do. This is designed so AI agents can read API docs and generate configs.

{
  "version": "1",
  "provider": "openai",
  "rules": [{
    "name": "chat-only",
    "endpoints": [
      {
        "method": "POST",
        "path": "/v1/chat/completions",
        "allow": true,
        "constraints": {
          "body": {
            "model": { "oneOf": ["gpt-4o-mini"] },
            "max_tokens": { "max": 1024 }
          }
        }
      },
      { "method": "GET", "path": "/v1/models", "allow": true }
    ],
    "rate_limit": {
      "requests_per_minute": 60,
      "requests_per_day": 1000
    },
    "deny_all_other": true
  }]
}

Constraint Types

ConstraintWhat it doesExample
oneOfField must be one of listed values"model": {"oneOf": ["gpt-4o-mini"]}
maxNumeric field must be ≤ value"max_tokens": {"max": 4096}
minNumeric field must be ≥ value"temperature": {"min": 0}
forbiddenField must not be present"stream": {"forbidden": true}

Path patterns support wildcards: /v1/files/* matches /v1/files/abc.

Provider templates are available at permissions/providers/ (OpenAI, Anthropic, GitHub).

9. Approval Workflow

When a placeholder key has Require Approval enabled:

  1. First request → proxy returns 403 duckway_approval_pending
  2. Pending approval appears in Approvals page (and notifications sent)
  3. Admin approves with a time window (e.g., 24 hours)
  4. Subsequent requests within the window proceed automatically
  5. After expiry, approval is needed again

This prevents unauthorized API usage even if a placeholder key is leaked — the admin must explicitly approve each client.

10. Key Groups & Load Balancing

A Key Group pools multiple API keys for a service. When a placeholder key points to a group, the server picks a key using the configured strategy:

StrategyBehavior
round-robinCycles through keys in order
least-usedPicks the key with lowest usage count
failoverUses first active key; falls back to next on failure

This helps avoid API rate limit outages by spreading requests across multiple keys.

11. Canary Tokens

Duckway integrates with canarytokens.org to deploy decoy credentials on client machines. If an attacker steals these and tries to use them, you get an email alert.

Setup

  1. Go to Canary Tokens page
  2. Enter your alert email
  3. Select which token types to deploy (default: all enabled)
  4. Optionally add client names to the exclude list
  5. Save — all new clients automatically get canary tokens on registration

Canary tokens are ON by default

Every new client gets canary tokens unless excluded. Use the exclude list for CI runners or test bots that shouldn't have decoys.

Available types (11)

API tokens (via canarytokens.org — real tripwire credentials):

  • AWS Credentials — fake AKIA key + secret in ~/.aws/credentials
  • GitHub Token — fake PAT in ~/.config/gh/hosts.yml.bak
  • Kubernetes Config — full kubeconfig in ~/.kube/config.bak
  • WireGuard Config — VPN config with private key

Local decoys (self-generated with embedded DNS canary):

  • .env File — mixed DB/API/webhook creds in ~/.env.production.bak
  • SSH Private Key — fake key in ~/.ssh/id_deploy
  • NPM Token — ~/.npmrc.bak
  • Docker Config — ~/.docker/config.json.bak
  • GCP Service Account — ~/.config/gcloud/application_default_credentials.json.bak
  • PyPI Token — ~/.pypirc.bak
  • Slack Token — bot + user tokens in ~/.config/slack/credentials.bak

All decoy files are NOT in keys.env — AI agents use real placeholder keys and ignore these.

13. CLI Reference

Server

# Start server
go run ./cmd/server/ --port 8080 --data /path/to/data

# Flags:
#   --port    Listen port (default: 8080, or DUCKWAY_LISTEN env)
#   --listen  Listen address (e.g., :8080, 127.0.0.1:9090)
#   --data    Data directory (default: ~/.duckway-server, or DUCKWAY_DATA_DIR env)

Client

duckway init          # Register with server (interactive)
duckway sync          # Fetch placeholder keys + canary tokens
duckway env           # Print keys as shell exports
duckway proxy         # Start local proxy on :18080
duckway proxy --port 9999  # Custom port
duckway status        # Show connection info + assigned keys
