1. Overview

Duckway is an API proxy that manages real API keys centrally. AI agents and other clients never see real keys — they use placeholder keys that look identical in format but are swapped for real keys by the proxy.

  • No key leakage — agents only have fake keys; real keys never leave the server
  • ACL system — 22 pre-built templates from official API docs + custom JSON rules
  • Approval workflow — require admin consent with configurable TTL, approve via Discord/Telegram
  • HTTPS MITM proxy — transparent interception with auto-generated TLS certificates
  • Key groups — round-robin, least-used, failover to avoid rate limits
  • 16 canary token types — auto-deployed honeypots detect compromised machines
  • Audit trail — filterable request log with client, service, key, status

3. Quick Start

Docker (recommended)

# Production
docker compose up -d

# Development (auto-seeds test data)
./scripts/dev.sh up

One-liner client install

curl -fsSL http://duckway-server:8080/install.sh | sh
duckway init

Admin setup

  1. Login — credentials shown in server log on first run (dev mode: duckway/duckway)
  2. Default services pre-configured: OpenAI, Anthropic, GitHub, Discord, Telegram
  3. API Keys → add your real keys (encrypted at rest)
  4. Clients → register a machine → copy the full setup command
  5. Assign keys to the client → placeholder keys auto-generated
  6. Optionally: set ACL, configure canary tokens, add notification channels

4. Services & API Keys

A Service = an upstream API. Each service configures:

NameSlug for proxy URL: /proxy/{name}/...
Upstream URLe.g., https://api.openai.com
Auth Typebearer, header, or query
Key Prefix + LengthFor placeholder format matching (e.g., sk-, 164 chars)
Key DirectoryWhere keys are deployed on client (e.g., ~/.config/openai/credentials)
Default ACLService-level access control — applies to all keys unless overridden

API Keys are your real credentials, stored AES-256-GCM encrypted. Each key can have its own ACL. Edit name, rotate the key value, or change ACL from the API Keys page.

5. ACL System

Three-layer access control. Request must pass ALL non-empty layers — each can only narrow, never widen:

  1. Service default_acl — widest scope, applies to all keys in service
  2. API Key acl — narrows further per real key
  3. Placeholder permission_config — narrowest, per client+service

Example: Service allows all OpenAI endpoints. Key restricts to chat + embeddings. Placeholder restricts to gpt-4o-mini only. An agent using this placeholder can only call chat completions with gpt-4o-mini — the intersection of all three layers.

Pre-built templates (22)

ServiceTemplates
OpenAIallow-all, chat-only, chat-embeddings, inference-all, no-admin
Anthropicallow-all, messages-only, no-batches
GitHuballow-all, read-only, repo-read, issues-prs, no-destructive, gists-only
Discordallow-all, webhook-only, messages-only, read-only
Telegramallow-all, send-only, read-only, no-admin

Custom ACL JSON

{
  "version": "1",
  "rules": [{
    "name": "chat-only",
    "endpoints": [
      {"method": "POST", "path": "/v1/chat/completions", "allow": true,
       "constraints": {"body": {"model": {"oneOf": ["gpt-4o-mini"]}, "max_tokens": {"max": 1024}}}},
      {"method": "GET", "path": "/v1/models", "allow": true}
    ],
    "rate_limit": {"requests_per_minute": 60},
    "deny_all_other": true
  }]
}

Constraints: oneOf (string match), max/min (numeric), forbidden (field must not exist). Path wildcards: /v1/files/*

6. Client Setup

Each machine registers as a Client with a unique name and 6-char short ID (e.g., k7x2m9). The short ID tags canary emails for per-machine alerting.

Registration

  1. Admin creates client → gets token + full setup command
  2. On the machine: curl -fsSL http://server/install.sh | sh && duckway init
  3. Client downloads CA cert, installs to system trust store
  4. Admin assigns API keys → placeholder keys generated
  5. duckway sync → keys in ~/.duckway/keys.env + canary files deployed

Per-client settings

Canary: ON/OFFToggle canary token deployment per client
Placeholder keysEach editable: env name, key path, approval, TTL
Canary tokensView deployed tokens, links to canarytokens.org management

8. Proxy Modes

Mode 1: HTTPS Proxy (transparent)

export HTTPS_PROXY=http://localhost:18080
curl https://api.openai.com/v1/chat/completions ...
# → duckway client (MITM) → server → key injection → upstream

Client generates per-host TLS certs signed by Duckway CA. Known service hosts are intercepted; unknown hosts tunnel through transparently.

Mode 2: Direct URL (lightweight)

curl http://duckway-server/proxy/openai/v1/chat/completions \
  -H "X-Duckway-Token: <token>" ...

No CA cert needed. Works as an OpenAI-compatible endpoint.

Heartbeat

Built-in /proxy/heartbeat/ping service. Every client auto-gets a heartbeat key. duckway status tests it.

9. Approval Workflow

  1. Agent request → 403 duckway_approval_pending
  2. Notification sent to Discord/Telegram with approval ID
  3. Admin approves: click ✅ reaction (Discord), tap button (Telegram), or !approve <id>
  4. Approval valid for the configured TTL (default 24h)
  5. After TTL expires → needs re-approval

TTL is per-placeholder, editable from the Clients page. Set to 0 to disable approval entirely.

13. CLI Reference

Server

duckway-server --port 8080 --data /path/to/data
# Env: DUCKWAY_LISTEN, DUCKWAY_DATA_DIR, DUCKWAY_DEV=1 (password=duckway)

Client

duckway init              # Register + download CA cert + sync keys
duckway sync              # Refresh keys + canary tokens
duckway env               # Print keys as shell exports
duckway proxy [--port N]  # Start HTTPS MITM proxy (default :18080)
duckway status            # Server, keys, heartbeat, local proxy, CA cert

Dev scripts

./scripts/dev.sh up       # Docker build + start + seed test data
./scripts/dev.sh restart  # Rebuild after code changes
./scripts/dev.sh down     # Stop (keep data)
./scripts/dev.sh nuke     # Stop + wipe data volume
./scripts/dev.sh logs     # Follow server logs
